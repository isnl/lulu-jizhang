# 记账小程序问题与优化分析

生成时间：2026-06-17  
范围：前端 Vue 应用、Cloudflare Pages Functions API、D1 数据结构、当前本地运行页面体验。  
说明：本分析只做只读评估，不涉及代码修改。

## 总体判断

这个小程序已经具备一个家庭记账工具的核心闭环：手工记账、多成员、按月/年统计、分类汇总、账单导入、重复识别、分类规则配置、API Token 管理。对个人/家庭使用来说，功能覆盖面不错，尤其是“导入账单后预览、调整分类、识别重复”的方向是对的。

目前主要瓶颈不在“缺功能”，而在三个方面：

1. 数据准确性依赖大量散落的规则，导入场景越多越容易出现漏判、误判。
2. 工程结构已经开始变重，导入组件和批量导入 API 承载了太多职责。
3. 安全配置和发布治理不够成熟，当前仓库里存在明文密钥/密码等高风险项。

建议优先把“账单导入与去重”整理成稳定的领域能力，再逐步优化 UI 和报表体验。

## 优先级概览

| 优先级 | 方向 | 主要收益 |
| --- | --- | --- |
| P0 | 移除明文密钥、修复类型检查、统一编码/分类常量 | 降低安全和维护风险 |
| P1 | 重构导入解析、去重、分类规则 | 提升账单导入准确性 |
| P1 | 增加导入审计与回滚能力 | 避免误导入后难以修复 |
| P2 | 优化统计表和明细交互 | 提升日常使用效率 |
| P2 | 增加测试样本库和回归测试 | 防止修一个账单又漏另一个账单 |
| P3 | UI 视觉收敛、移动端优化、性能优化 | 提升使用舒适度 |

## 一、数据准确性问题

### 1. 重复识别规则分散且不可解释

当前重复识别逻辑同时存在于前端导入预览和后端批量导入中：

- `src/components/ImportBill.vue`
- `functions/api/records/batch.ts`

这会带来两个问题：

- 前后端规则容易不一致。预览里显示可导入，提交时可能被后端跳过；或者预览没识别重复，提交后才合并。
- 每遇到一个新账单漏判，就需要继续追加规则，规则复杂度会线性膨胀。

建议：

- 抽出统一的 `dedupe` 领域模块，至少把规则定义、文本标准化、source 归一化、日期容差、匹配原因统一。
- 每条重复规则返回明确原因，例如：
  - `same_source_transaction_id`
  - `same_salary_income`
  - `bank_credit_transfer_same_counterparty`
  - `bank_third_party_payment`
- 预览弹窗展示“为什么判为重复”，方便人工确认。

### 2. 导入解析强依赖格式和启发式规则

当前支持微信、支付宝、京东、银行、信用卡/PDF，但解析逻辑集中在 `ImportBill.vue` 这个单文件中。这个文件约 1900 行，已经是一个明显的复杂度信号。

风险：

- 各平台账单格式稍有变化就可能解析失败。
- PDF 解析有 AI 和本地正则两条路径，结果稳定性不易保证。
- 账单来源字段、交易号、备注、对方户名等字段有时为空，影响去重。

建议：

- 按来源拆分解析器：
  - `parsers/wechat`
  - `parsers/alipay`
  - `parsers/jd`
  - `parsers/bank`
  - `parsers/credit`
- 统一输出一个标准 `ImportedRecord`，包含：
  - `source`
  - `sourceTransactionId`
  - `counterparty`
  - `product`
  - `rawText`
  - `dedupeHints`
  - `confidence`
- 保存原始账单文件名和导入批次 ID，后续排查会轻松很多。

### 3. 缺少导入批次概念

目前记录是直接落在 `records` 表中，导入成功后很难回答：

- 这批账单导入了哪些记录？
- 哪些被跳过？
- 哪些重复项被合并？
- 如果导入错了，怎么撤销？

建议新增导入批次模型：

- `import_batches`
  - id
  - source
  - file_name
  - member_id
  - imported_count
  - skipped_count
  - created_at
- `records.import_batch_id`
- 预览页显示批次摘要，导入后支持“查看本次导入结果”和“撤销本次导入”。

### 4. 分类体系重复定义，容易漂移

分类常量在多个地方重复出现：

- `src/types/index.ts`
- `functions/api/records.ts`
- `functions/api/records/batch.ts`
- `functions/api/bill/analyze.ts`
- `schema.sql`

风险：

- 前端允许的分类、后端校验、数据库 CHECK 约束可能不一致。
- 新增分类时需要改很多地方，容易漏。

建议：

- 后端以数据库表或单一配置文件作为分类真源。
- 前端通过 API 获取分类列表。
- 数据库避免硬编码过长 CHECK 枚举，改用分类表 + 外键或应用层校验。

## 二、产品体验问题

### 1. 首页统计表信息密度高，但缺少“洞察层”

当前主页面是宽表：月份 x 分类。对于核账很有用，但对日常使用来说需要用户自己找异常。

建议增加几个顶部洞察：

- 本月支出环比变化。
- 支出最高的 3 个分类。
- 本月异常大额支出。
- 重复/疑似重复记录待处理数量。
- “工作待报销”未报销合计。

这样用户打开页面后能直接知道“该关注哪里”。

### 2. 明细查看入口隐藏在表格单元格点击中

当前通过点击统计表格里的金额进入明细，功能有效，但发现成本较高。

建议：

- 金额单元格增加更明确的 hover/icon 提示。
- 表格上方增加“查看全部明细”入口。
- 明细弹窗支持按来源、成员、金额区间、关键词组合过滤。
- 明细里显示重复来源、导入批次、原始交易号。

### 3. 导入预览还可以更像“审核工作台”

当前导入预览已经有“需导入”和“可能重复”两块，但对批量审核仍略弱。

建议：

- 重复项按原因分组：同交易号、同工资、同转账对象、第三方代扣等。
- 支持批量操作：
  - 全部跳过
  - 全部恢复导入
  - 仅导入某成员
  - 仅导入某分类
- 预览中显示“匹配置信度”。
- 每条记录保留“原始文本”展开查看。

### 4. 浏览器原生 `confirm/alert` 体验割裂

多处组件使用原生弹窗：

- 删除记录
- 删除 Token
- 删除分类规则
- 禁用成员

建议统一为应用内确认弹窗，样式、文案、按钮状态保持一致。对于删除记录这类不可恢复操作，可以展示记录摘要，降低误删风险。

## 三、工程质量问题

### 1. 单文件复杂度偏高

当前几个核心文件较大：

- `src/components/ImportBill.vue`：约 1910 行
- `src/components/RecordTable.vue`：约 794 行
- `functions/api/records/batch.ts`：约 642 行
- `functions/api/bill/analyze.ts`：约 316 行

这说明业务正在堆叠到少数文件里。短期能跑，长期会让修改成本越来越高。

建议拆分：

- `ImportBill.vue` 只保留 UI 和流程状态。
- 解析逻辑放入 `src/domain/import/parsers/*`。
- 去重逻辑放入共享的 `src/domain/dedupe` 或后端 API 中。
- 分类逻辑放入 `src/domain/category`。
- API 侧复用同一套规则，避免前后端复制。

### 2. 类型检查当前不可用

执行 `npm run type-check` 会失败：

```text
Search string not found: "/supportedTSExtensions = .*(?=;)/"
```

这通常是 `vue-tsc` 与 TypeScript 版本兼容问题。类型检查不可用意味着很多模板/类型问题无法在提交前发现。

建议：

- 升级 `vue-tsc` 到与当前 TypeScript 兼容的版本，或锁定 TypeScript 版本。
- 把 `type-check` 纳入提交前或部署前检查。
- 后续再加最小测试集。

### 3. 编码/字符集状态混乱

终端读取时多处中文显示为乱码，代码中也可见一些疑似乱码字符串。这会影响：

- 分类常量可读性。
- 错误提示维护。
- 正则规则可维护性。
- 数据库 schema 可读性。

建议：

- 全仓库统一 UTF-8。
- 对历史乱码文本做一次审计修复。
- 尽量避免在代码里到处硬编码中文分类和提示，改为集中配置。

### 4. 缺少自动化测试，尤其是导入回归样本

当前最容易出错的是账单导入和重复识别，但这恰恰最需要测试。

建议建立样本库：

```text
tests/fixtures/
  wechat/
  alipay/
  jd/
  bank/
  credit/
```

每个样本保存：

- 原始文件或脱敏文本。
- 预期解析记录数。
- 预期分类。
- 预期重复匹配结果。

优先覆盖最近已经暴露的问题：

- 张欢 1000 / 2675 / 2728 转账重复。
- 2026-01-19 工资收入 12449.02。
- 银行流水与信用卡账单互相补充。
- 支付宝/微信/银行卡第三方代扣。

## 四、安全和部署风险

### 1. 明文密钥和管理员密码在配置文件中

`wrangler.toml` 里存在：

- `DEEPSEEK_API_KEY`
- `JWT_SECRET`
- `ADMIN_USERNAME`
- `ADMIN_PASSWORD`

这是最高优先级风险。即便是个人项目，也建议立即迁移到 Cloudflare secret 或本地 `.dev.vars`，并轮换已经暴露的密钥。

建议：

- 生产环境使用 `wrangler secret put`。
- 本地使用 `.dev.vars`，且不要提交。
- 立即轮换 DeepSeek API Key、JWT Secret、管理员密码。
- `wrangler.toml` 只保留非敏感配置。

### 2. CORS 过宽

当前 `Access-Control-Allow-Origin: *`。如果后续部署到公网并使用 Authorization 或 API Token，建议收敛来源。

建议：

- 生产环境只允许自己的域名。
- API Token 入口可以单独设计跨域策略。
- 对高风险接口增加来源和速率限制。

### 3. API Token 明文存储风险

从代码看 API Token 直接用 token 查询。若数据库泄露，Token 可直接使用。

建议：

- 存储 token hash，而不是原文。
- 创建 token 时只展示一次原文。
- 支持 Token 权限范围和过期时间。

### 4. PDF 文本被输出到浏览器控制台

`ImportBill.vue` 中会 `console.log(text)` 输出 PDF 全文。账单属于敏感财务数据，不建议在控制台保留。

建议：

- 开发环境可选 debug。
- 生产环境禁止输出账单全文。
- 出错时只输出文件名、页数、解析统计，不输出交易明细。

## 五、数据模型与报表能力

### 1. `records` 表已经开始承载太多隐含语义

当前记录表包括基础记账字段、导入来源、交易对方、商品、支付方式、去重 key、成员等。后续如果继续扩展，建议拆出更多结构：

- 导入批次。
- 原始交易表。
- 去重匹配表。
- 分类变更历史。

### 2. 缺少变更历史

现在支持修改分类和备注，但看不到：

- 谁改了。
- 从什么改成什么。
- 为什么改。

家庭记账中，分类调整很常见。建议增加简单审计表：

- record_id
- field
- old_value
- new_value
- changed_at

### 3. 报销、还贷、内部转账可以更结构化

目前“工作待报销”“还贷”“生活费转账”主要靠分类和备注表达。建议引入状态字段：

- `reimbursement_status`: pending / reimbursed / ignored
- `transfer_type`: internal / external / loan / living_expense
- `linked_record_id`: 用于一进一出或跨来源配对

这样统计时可以选择是否排除内部转账，家庭净支出会更准确。

## 六、性能与可用性

### 1. 大表横向滚动可用，但移动端压力较大

分类多时表格很宽。桌面核账没问题，手机上会比较累。

建议：

- 移动端改成分类卡片或可折叠列表。
- 默认展示 Top 分类，其余折叠。
- 支持“只看有金额的分类”。

### 2. 导入大文件时缺少阶段反馈

现在导入过程主要是 loading。实际流程包括读文件、解析、AI 分析、拉历史、去重、生成预览。

建议显示阶段：

1. 读取文件
2. 解析账单
3. 匹配成员
4. 检查重复
5. 生成预览

失败时给出更具体的错误位置。

### 3. AI 解析应有成本和失败降级策略

PDF 账单调用 DeepSeek。建议：

- 优先走确定性解析。
- AI 只作为 fallback 或补充。
- 给 AI 请求加超时、重试、错误提示。
- 记录 AI 解析版本和 prompt 版本，便于追踪差异。

## 七、建议路线图

### 第一阶段：止血和稳定

目标：降低安全风险，保证现有功能可维护。

- 移除并轮换明文密钥和密码。
- 修复 `npm run type-check`。
- 删除生产环境 PDF 全文 console 输出。
- 整理分类常量，减少重复定义。
- 给导入去重规则加最小测试样本。

### 第二阶段：导入系统重构

目标：让账单导入准确、可解释、可回滚。

- 拆分各账单解析器。
- 建立统一 `ImportedRecord`。
- 后端提供 `/api/import/preview`，前端不再复制去重规则。
- 增加导入批次和导入结果页。
- 增加“重复原因”和“置信度”。

### 第三阶段：产品体验升级

目标：从“能看账”变成“帮你发现问题”。

- 首页增加支出洞察。
- 明细页增加多条件过滤。
- 支持报销状态和内部转账排除。
- 导入预览变成审核工作台。
- 移动端改为卡片/折叠视图。

### 第四阶段：长期治理

目标：让项目能持续演进。

- 建立导入 fixtures 回归测试。
- 建立 API 单元测试。
- 加入基础 CI：build、type-check、lint、关键测试。
- 数据库迁移版本化，避免直接修改主 schema。

## 最值得优先做的 5 件事

1. 轮换并移除 `wrangler.toml` 中的明文密钥和密码。
2. 修复 `vue-tsc` 类型检查，让 `npm run type-check` 可用。
3. 把导入去重逻辑从前端和后端复制状态中抽出来，至少先统一规则定义。
4. 建立账单导入回归样本，覆盖最近出现的漏判账单。
5. 增加导入批次和撤销能力，降低误导入成本。

## 结论

这个小程序的方向是扎实的，已经不只是“记一笔”的玩具应用，而是一个面向真实家庭账务的工具。现在最需要的是把“真实账单世界的复杂性”收束成稳定的工程结构：解析、分类、去重、审核、落库、回滚各司其职。

如果下一步只做一个大方向，我建议优先做“导入系统产品化”：统一预览 API、导入批次、重复原因、回归测试。这会直接解决你最近连续遇到的漏识别问题，也会让后续新增银行/账单格式不再那么费劲。
